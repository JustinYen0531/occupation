# 不是因為它簡單，而是我選擇讓它變簡單

在程式設計的圈子裡，流傳著一張被印在紅色橫幅上的經典迷因圖。它改造了美國前總統甘迺迪那段著名的登月演講：

「我們決定去做這件事，不是因為它很簡單，而是因為我們『以為』它會很簡單。」  
（*“We do this not because it is easy, but because we thought it would be easy.”*）

第一眼看到這句話時，我差點笑出聲。這簡直是所有開發者最真實的寫照——我們總是過度樂觀地低估任務的複雜度，滿懷信心地跳進坑裡，最後在截止日前一邊灌著能量飲料一邊瘋狂 debug。

然而，在經歷了這十幾週高強度的開發期，在期中考、雙主修學業與多個競賽專案之間多線作戰後，我對這句玩笑話有了完全不同的體悟。對於我這種容易被巨大任務壓垮、卻又對作品品質有著偏執要求的人來說，這句話不再只是一句自嘲，而是一個被我奉為圭臬的開發心法：**既然任務本身不可能變簡單，那我就必須主動選擇，讓它「看起來」變簡單。**

---

## 我不是突然變強，而是學會了「先開始」

老實說，我以前是個典型的「計畫型拖延症患者」。每當要開始一個新專案，我的大腦就會不由自主地開始描繪最終版本那無比宏大、完美的模樣。我看著腦海中那個精美絕倫、功能無懈可擊的終極成品，再回頭看看眼前這片空無一物的程式碼編輯器，巨大的落差感與壓力排山倒海而來，壓得我根本不知道第一行程式碼該從哪裡寫起。

這種被目標「嚇傻」的狀態，直到我開始開發一款將股市分析與滑雪遊戲結合的金融教育遊戲時，才有了轉變。

這款遊戲的構想是將枯燥的股市分析儀表板轉化為滑雪主題的樂園冒險，玩家可以一邊在雪道上前進、一邊學習如何看懂股票的走勢。如果在一開始就去思考如何渲染高細節的 3D 地形、如何對接即時股價 API，我大概又會退縮。但這一次，我強迫自己「被動照做」——先跟著工作坊的基礎教學，老老實實地搭建出一個只有最基本介面、甚至有點簡陋的財務儀表板原型。

我對自己說：「沒關係，這版很爛，但它能跑。」

當這個粗糙的原型在瀏覽器上順利運作的那一刻，我突然發現，接下來要往上疊加功能、調整滑雪角色的物理反饋、甚至微調鏡頭跟隨的流暢度，都變得沒有想像中那麼難了。因為最難的「從零到一」已經過去了。

在後來開發一款課堂互動的大富翁遊戲時，我也面臨了類似的處境。作為第一組要在課堂上演示的團隊，時間壓力大到讓人窒息。但我意識到，與其把 deadline 當作在背後追趕的怪物，不如主動出擊，在還有餘力時先架設出最核心的單頁應用（SPA）框架和空白題庫。當最基本的骨架搭建好，deadline 的焦慮感自然減輕了。我學會了：**克服開始焦慮的唯一解藥，就是做一個哪怕再不成熟、但可以運作的垃圾原型。**

## AI 協作：將抽象的高牆拆成可操作的階梯

在這段密集開發的日子裡，AI 成了我形影不離的協作夥伴。但我逐漸明白，AI 對於開發者的價值，絕對不是「直接幫我把程式碼寫完」，而是扮演一個「降噪器」與「任務拆解器」，幫我降低進入創作狀態的門檻。

在開發一款結合醫療數據標準與情感陪伴的 AI 輔助工具時，我第一次接觸到了複雜的醫療數據交換格式（FHIR）這個龐大且充滿抽象術語的領域。一開始，我被各種技術規範砸得頭昏腦脹，甚至產生了放棄參賽的念頭。

於是我把這些抽象的概念丟給 AI，並一遍又一遍地追問：「請用更簡單、更人性化的語言解釋這段話。」「如果我要做一個最簡單的資料上傳功能，第一步應該是什麼？」

AI 就像一個極有耐心的導師，把那些高深莫測的術語拆解成我可以理解的白話文，並幫我指出了一條清晰的起跑線。

到了專案後期的決賽衝刺階段，時間只剩下短短五天。面對堆積如山的需求——從醫生端工作區、雙重角色認證，到抑鬱量表評估和雲端部署穩定性——我的大腦再次陷入當機狀態。這一次，我沒有硬撐，而是坐下來和 AI 進行了一場「會議」。我把所有腦海中凌亂的想法一股腦地寫成文檔，請 AI 幫我分析哪些是「核心功能（Must-have）」，哪些是「展示用（Nice-to-have）」，並按照優先順序和開發難易度，幫我整理出一張條理分明的待辦清單（Checklist）。

看著那張被拆解成一個個「15 分鐘就能完成的小任務」的清單，我突然覺得，這堵高牆似乎也沒那麼難爬了。這讓我看清了開發的本質：**資深工程師與普通開發者的真正差距，往往不在於寫程式的速度，而在於將複雜問題拆解為「可控小模組」的工程思維。**

## 真正成熟的開發，是學會溫柔的取捨

然而，並非每一次的實踐都是一帆風順的。在學期中考的地獄週，我的生活差點失去了平衡。

那週我同時面臨了醫療陪伴 AI 輔助工具的決賽答辯準備、課堂互動大富翁遊戲的正式上線演示，以及微積分等學科的期中考試。因為對自己作品的高標準，我不肯放過任何一個細節，試圖把所有事情都做到完美。結果就是，我差點因為嚴重的睡眠不足而在考場上崩潰，甚至在考前祈禱奇蹟發生（幸好最後考試沒考到我沒讀的那一章，這大概是命運對我的仁慈）。

那次經歷給了我一記重拳，也讓我重新思考了「完美」的定義。

在有限的時間與資源下，追求絕對的完美只會把事情推向更複雜、更不可控的深淵。真正成熟的開發，不是無止境地往上堆砌功能，而是學會妥協，並在正確的時機按下暫停鍵。

在醫療陪伴專案的系統重構過程中，我曾為了提升效能，忍痛扔掉了一整套自己花了幾天設計、但與新架構不相容的 AI 工作流。那時候心真的很痛，但我知道，不捨棄它，專案就無法前進。在做一款旨在解決租屋資訊不透明的風險評估工具時，面對現實世界中不完整、缺失的欄位數據，我不再強求完美清洗，而是設計了容許缺失值的機制，這何嘗不是一種對不完美現實的優雅妥協？

同時，這個租屋工具的專案也打破了我一直以來「單打獨鬥」的習慣。因為比賽規定必須三人以上組隊，我被迫走出舒適圈去招募隊友。以前我總覺得「自己做比較快」、「問隊友很麻煩」，但在這場跨領域的協作中，我發現當我拋出一個「已經有具體原型」的構想時，隊友們其實非常樂意加入並幫忙。AI 在技術上或許比某些人更有效率，但「學習如何與人類協作、如何發揮領導力」，卻是 AI 無法代勞、卻也最迷人的成長過程。

到了最後一個關於災害應變的系統原型專案時，我驚訝地發現自己已經不再像以前那樣慌亂。面對全新的競賽題目，我沒有急著去刻 UI 畫面，而是先花了一整天的時間，慢條斯理地和隊友釐清數據交換的 JSON 格式、定義好行動裝置的使用者經驗預留框。我們把規則和資料格式徹底想清楚了，才動手寫第一行 code。這種先釐清規格（Clarify the spec first）的 SOP，正是這十幾週來，座右銘在我的開發日常中留下的最深烙印。

## 這句話現在對我的意義

回頭看，這句原本像是在開玩笑的座右銘，如今已經內化成我面對生活與開發的底氣。

“We do this not because it is easy, but because it would be easy.”

它不是叫我們盲目樂觀地去否認事情的困難，而是提醒我們：**當我們面對一個看似高不可攀的任務時，我們有權力選擇不再被它嚇倒。我們可以用規劃、用拆解、用 AI 協作、用清單、用優先順序的取捨，把這個困難的任務，重塑成一條比較容易靠近、比較容易開始的道路。**

但同時，我也在這段旅程的終點，看到了自己的極限。

看著桌旁堆積如山的提神飲料空罐，回想起無數個為了專案而顛倒黑白的夜晚，我知道這樣的燃燒模式是無法長久的。用犧牲睡眠和健康換來的精緻作品，終究只是一種短期爆發的狂熱。

一個真正走得長遠的創作者，必須學會控制自己的開發密度。我依然熱愛把腦海中的瘋狂點子變成現實，但我希望未來的自己，能在創作的熱情與健康的作息之間，找到一個更穩定、更溫柔的平衡點。畢竟，我們之所以選擇讓開發變簡單，不就是為了讓自己能更長久、更快樂地創作下去嗎？

### 作者後記
這篇文章的靈感與素材，整理自我在這學期核心通識課程中所寫的英文個人座右銘實踐報告。當時只是為了每週交差而記錄下自己的 commit 與心得，但當學期步入尾聲，回頭審視這段瘋狂的日子時，才驚覺自己默默走過了這麼多心態上的坎。我不希望這些真誠的反思只停留在作業繳交的系統裡，因此花了一些時間將它重新用中文整理成這篇部落格文章，並隱去了那些繁瑣的專案代號。希望這篇帶著一點焦慮、自嘲與成長的紀錄，能給同樣在創作路上感到迷茫、或常常被大型專案嚇到的你，帶來一點點開始的勇氣。
